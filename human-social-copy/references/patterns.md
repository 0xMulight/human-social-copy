# 参考模板

用户需要模板、示例、多版本文案时，再使用这里的内容。

## 基础公式

第一行：

直接说出读者在意的结果、风险、误区或机会。不要夸张，不要绕。

中间：

给3到7个有用点。每一点都要让读者知道怎么判断、怎么做，或者怎么少踩坑。

结尾：

只放一个动作。比如评论关键词、收藏、关注后续更新、转给需要的人，或者说出自己的情况。

## 第一行选择

第一行优先写“读者看完或照做之后会有什么变化”。

优先级：

1. 结果变化：从A到B、少做某件事、多得到某个能力。
2. 具体场景：适合谁、在哪个环节用、解决什么问题。
3. 风险提醒：哪里容易踩坑、哪里容易浪费时间。
4. 清单价值：整理了哪些工具、步骤、标准或链接。
5. 个人判断：我会怎么选、怎么试、怎么放进工作流。

保留强钩子的原则：

- 原稿第一行已经有结果感时，不要改成普通提醒。
- 原稿里有具体对象、场景、数字或链接时，优先保留。
- 可以去掉夸张词，但不要去掉读者最关心的变化。
- 如果有两个开头，一个有结果变化，一个只是建议，优先选有结果变化的。

示例：

较弱：

```text
Codex要用得顺，先别急着装一堆插件。
```

更强：

```text
Codex想从“能写代码”升级到“能参与开发工作流”，这几个GitHub高星项目可以先收藏。
```

更克制但仍保留结果感：

```text
Codex想从写代码升级到参与开发流程，先补这几类工具。
```

## 选题判断

好选题通常至少满足一条：

- 对读者有用：教程、工具、清单、避坑、案例、数据整理。
- 贴近当下热度：市场正在讨论、项目刚更新、规则刚变化、AI模型刚更新、AI工具刚发布、新AI玩法正在被讨论、很多人正在问。

如果草稿只是泛泛观点，先加一个具体例子，或者加一个小清单，再写最终文案。

## 常用开头

可以从这些方向选一个：

- 结果型：想提高互动，先把第一行写清楚。
- 变化型：Codex想从写代码升级到参与开发流程，先补这几类工具。
- 经验型：我试了几次后，发现这个方法更稳。
- 避坑型：很多人卡住，问题常出在最后一步。
- 时机型：这个项目现在可以先观察，别急着重投入。
- AI型：这个AI工具值得试，但先别急着替换原来的工作流。
- 清单型：看空投项目，我会先查4件事。

避免假紧迫、假确定、假震惊。

## AI内容文案

结构：

1. 第一行写清这个AI工具、模型或玩法为什么值得看。
2. 说明它解决了什么具体问题。
3. 写出真实使用场景、限制和适合的人。
4. 给出个人判断，不要把体验写成官方结论。
5. 结尾放一个轻CTA。

示例骨架：

```text
这个AI工具可以试，但别一上来就换掉原来的流程。

我会先看三件事：

1. 它是不是真的省时间
2. 输出结果能不能稳定复用
3. 出错后好不好改

如果只是偶尔惊艳，不一定值得长期用。能稳定帮你少做重复活，才更适合放进工作流。

想看我按这个标准拆AI工具，可以评论工具名。
```

## 工具清单文案

结构：

1. 第一行写清这个清单能让读者获得什么变化。
2. 说明筛选标准，避免像单纯堆链接。
3. 每个工具只写最适合的使用场景。
4. 给出组合建议或先后顺序。
5. 结尾只引导收藏、尝试或评论。

示例骨架：

```text
Codex想从写代码升级到参与开发流程，先补这几类工具。

我筛工具时主要看三件事：

1. 能不能解决真实开发里的高频问题
2. 项目是不是还活跃
3. 接进Codex后是不是能马上用起来

个人开发可以先看GitHub MCP、Context7和Serena。前端项目再加Playwright MCP。资料抓取多，再看Firecrawl。

先收藏这份清单，后面配Codex时可以对照试。
```

## cryptoYap文案

结构：

1. 第一行写清关注点。
2. 说明为什么现在值得看。
3. 列出需要检查的点，比如融资、产品、用户活跃、任务成本、规则、风险。
4. 给出个人判断，但语气要留有余地。
5. 结尾放一个轻CTA。

示例骨架：

```text
这个项目可以先放进观察列表，别急着重投入时间。

我会先看三件事：

1. 任务有没有稳定更新
2. 交互成本会不会太高
3. 官方有没有清楚说明积分或资格规则

目前更适合小成本测试。后面规则更清楚，再加时间也不晚。

想看我按这个标准拆项目，可以评论项目名。
```

## 改写检查

删除：

- 空泛使命表达
- 没有证据的夸张判断
- 不增加信息的漂亮词
- 重复小标题
- 括号
- 太多标签
- 英文单词或缩写前后的多余空格
- 有用内容前面的长铺垫

保留：

- 原稿里最强的第一行
- 用户给出的具体数字
- 真实限制
- 个人过程
- 清楚的下一步动作
- 风险提醒
