# 自适应开头钩子

这个文件用于帮助Agent根据文章内容自动选择开头钩子。重点是先判断内容类型、目标读者和正文能交付的价值，再选择最适合的开头模型。

不要直接照搬夸张句式。可以学习结构，但最终文案要符合human-social-copy的语气：具体、克制、可信。

## 总原则

钩子允许有冲突感、结果感、反差感，但不能靠空泛夸张撑场面。

第一句必须让读者立刻知道三件事：这篇内容和我有什么关系、能解决什么问题、为什么值得继续看。

判断方式：

- 冲突感来自真实矛盾，不来自硬造对立。
- 结果感来自正文能交付的变化，不写正文没有的承诺。
- 反差感要有过程、数据、案例或真实经验支撑。
- 如果一句话只剩情绪词，没有具体对象、问题或收益，删掉重写。
- 开头避免泛泛归因和模板反差，尤其是“很多人...”“别只当成...”“不是...而是...”。
- 解释概念时避免“简单讲”“简单理解”“简单说”，直接写对象、用途和场景。

## 基本流程

写第一行前，先提取6个信息：

1. 文章类型：教程、清单、避坑、案例、复盘、观点、工具推荐、项目分析。
2. 目标读者：新手、创作者、开发者、运营、投资者、项目方、某个具体人群。
3. 核心收益：省时间、省钱、少踩坑、提高效果、学会方法、看懂趋势。
4. 核心问题：读者现在卡在哪里、担心什么、误解什么。
5. 可信证据：真实经历、数据、步骤、工具名、项目名、截图、链接、对比结果。
6. 情绪强度：提醒、好奇、担心、惊喜、反差、收藏价值。

然后生成2到3个开头，分别用不同钩子模型。最后选最具体、最可信、信息密度最高的一句。

## 开头禁区与改写法

以下开头只适合当负面样本，默认不要用：

- 泛泛归因：很多人、大家都、大多数人。
- 虚设误区：别只当成、别只把、别再以为。
- 排比反差：不是...而是...、不是普通X，更像Y、X不在...而在...。
- 解释垫话：简单讲、简单理解、简单说。
- 无证据夸张：99%的人、千万不要、全网都在。

改写时直接给对象、场景、变化或判断标准。

```text
弱：很多人把Arc House当成普通社区，其实不是。
强：Arc House把builders、Office Hour和生态资源聚在一个入口里，适合想长期参与Arc的人先看。
```

```text
弱：别只把agentic wallet当成钱包。
强：agentic wallet解决的是AI Agent如何在规则约束下持有和使用USDC。
```

```text
弱：Arc House不是普通社群，更像Arc的生态前台。
强：Arc House把社区活动、builders交流和生态资源放在同一个入口，适合想长期参与Arc的人先看。
```

```text
弱：简单讲，Arc House就是Arc的Discord。
强：Arc House更像Arc给builders准备的社区入口，里面承接Office Hour、项目交流和生态资源。
```

## 黄金位置

最强信息放在第一行前10到15个字里。

优先放：

- 读者身份：做AI内容的人、刚开始用Codex的人、前端开发者。
- 核心问题：总写旧API、PR看不过来、前端验证太慢。
- 结果变化：从写代码到参与开发流程、从0到1跑通、少翻几十个页面。
- 数字承诺：3步、4类工具、8个检查点。

不要把最有用的信息藏在第二段。

## 钩子模型

### 1. 好奇模型

结构：只说一半的发现加关键留白。

适合：文章里有反常识发现、隐藏用法、容易被忽略的方法。

```text
我最近发现，让Codex更好用的关键不在提示词。
```

```text
MCP装完没效果时，先检查权限和调用场景。
```

慎用：不要为了好奇故意卖关子。下一句必须给具体信息。

### 2. 痛点模型

结构：精准指出卡点加立刻给方向。

适合：教程、问题解决、避坑、工具配置。

```text
Codex总写旧API？先让它读当前版本文档。
```

```text
PR看不过来时，我会先让Codex查这3类信息。
```

注意：问题要具体，不要只写“效率低”“很麻烦”。

### 3. 反差模型

结构：过去状态加现在结果。

适合：个人复盘、案例、方法验证、数据变化。

```text
以前我让AI改文案只会变平，现在会先保留强钩子。
```

```text
从复制日志到直接查CI，Codex接上GitHub后差别很明显。
```

注意：反差必须有真实过程或证据支撑。

### 4. 圈定受众模型

结构：点名一类人加明确提醒。

适合：垂直人群内容、教程、避坑、工作流建议。

```text
刚开始用Codex的人，先从GitHub和Context7这类高频工具补起。
```

```text
做前端验证的人，可以优先试Playwright MCP。
```

注意：受众要具体。不要动不动写“所有人”。

### 5. 风险提醒模型

结构：指出潜在风险加给预防方法。

适合：AI工具、crypto、空投、安全、账号、职业、成本相关内容。

```text
让Agent接GitHub前，先把权限范围想清楚。
```

```text
空投任务先算交互成本和规则风险，再看奖励。
```

注意：提醒可以有压力，但不要恐吓。

### 6. 后果模型

结构：放大真实坏结果加给处理方法。

适合：读者真的可能损失时间、钱、账号、机会的内容。

```text
MCP权限开太大，后面排查问题会很麻烦。
```

```text
只看项目热度就重投入，很容易把时间花错地方。
```

注意：只有存在真实后果时才用，不要虚构风险。

### 7. 高价值模型

结构：强调稀缺信息或可收藏价值。

适合：资源合集、工具清单、配置组合、步骤说明、判断标准。

```text
Codex插件先按工作流缺口装，这套组合可以先收藏。
```

```text
我会按这4个标准筛Codex插件。
```

注意：高价值要靠内容密度支撑，不要只喊“干货”。

### 8. 借势模型

结构：借可信对象、项目、社区共识或真实案例引出观点。

适合：有明确来源、官方项目、明星工具、社区讨论的内容。

```text
GitHub官方MCP出来后，Codex能做的事多了一截。
```

```text
Playwright接进Agent后，前端验证可以直接跑交互和截图。
```

注意：不能编造“专家都在用”“大佬推荐”。没有来源就不用。

### 9. 数字清单模型

结构：数字加明确收益。

适合：步骤、清单、工具合集、检查表、对比项。

```text
让Codex更像开发助手，我会先补这4类能力。
```

```text
接入GitHub MCP前，先检查这3个权限点。
```

注意：数字后面要真的有对应内容。

### 10. 省时省钱模型

结构：节省成本加具体方法。

适合：工作流、工具、教程、自动化、资源选择。

```text
少翻几十个GitHub页面，可以先让Codex查这几类信息。
```

```text
Repomix适合把仓库整理成AI能快速读取的上下文。
```

注意：不要承诺无法证明的效率提升。

### 11. 新手友好模型

结构：新手身份加最小可执行步骤。

适合：入门教程、配置、工具第一次使用。

```text
第一次给Codex接MCP，先从GitHub和Context7开始。
```

```text
新手配Agent工具，先补最常用的两三个工作流缺口。
```

注意：给新手的开头要降低门槛，不要堆术语。

### 12. 亲测体验模型

结构：我试过加个人判断。

适合：真实体验、复盘、工具评测、方法对比。

```text
我试了几类Codex插件，真正高频用到的是这几种。
```

```text
这套文案规则改完后，最明显的变化是开头不容易被洗平。
```

注意：没有亲自试过就不要写“亲测”。

### 13. 误区纠偏模型

结构：常见误解加正确做法。

适合：观点纠偏、避坑、教程、方法论。

```text
Codex插件装得多不一定更好，先补最常用的工作流缺口。
```

```text
写开头先看文章真正能交付什么结果，再决定要不要加冲突。
```

注意：不要使用“你错了”这类攻击性表达。

## 内容类型到钩子模型

- 教程：痛点模型、数字清单模型、新手友好模型。
- 工具清单：高价值模型、数字清单模型、结果变化模型。
- 避坑：痛点模型、风险提醒模型、误区纠偏模型。
- 案例复盘：反差模型、亲测体验模型、结果变化模型。
- 观点分析：好奇模型、反差模型、误区纠偏模型。
- AI工具体验：痛点模型、亲测体验模型、借势模型。
- AI工作流：结果变化模型、痛点模型、数字清单模型。
- crypto或空投：风险提醒模型、成本模型、时机模型。
- 资源合集：高价值模型、数字清单模型、圈定受众模型。

## 组合使用

一个开头可以融合2种模型，但不要堆太多。

常用组合：

- 痛点加方案：Codex总写旧API？先让它读当前版本文档。
- 结果加数字：让Codex更像开发助手，我会先补这4类能力。
- 圈定受众加提醒：刚开始用Codex的人，先从GitHub和Context7这类高频工具补起。
- 反差加证据：从复制日志到直接查CI，GitHub MCP解决了一个高频麻烦。
- 高价值加收藏：Codex插件先按工作流缺口装，这套组合可以先收藏。

## 输出要求

生成开头时，先在心里判断模型，不要把模型名写出来。

最终只输出自然文案，不要写“我选择痛点模型”。

如果用户要求多个版本，可以给3个方向：

1. 稳重版：更像真实经验分享。
2. 抓人版：信息更前置，但不夸张。
3. 垂直版：更明确点名目标读者。

## 不要这样写

- 不要滥用“99%不知道”“救命”“封神”“绝了”“狠狠爱了”。
- 不要编造“大佬都在用”“全网都在推”“官方暗示”。
- 不要用“很多人...”这类泛泛开头代替具体人群和具体场景。
- 不要用“别只当成...”这类虚设误区代替真实判断。
- 不要用“不是...而是...”这类排比反差代替具体变化。
- 不要用“简单讲”“简单理解”“简单说”代替具体解释。
- 不要只写情绪，不给信息。
- 不要把文章里没有的结果写进第一行。
- 不要把所有内容都套成恐惧或焦虑。

## 开头自检

输出前检查：

- 第一行是否能单独成立。
- 第一行是否和正文内容一致。
- 第一行是否把最强信息放在前10到15个字。
- 第一行是否能让读者知道看下去能得到什么。
- 第一行是否具体到某个场景、问题、收益或人群。
- 第一行是否避开“很多人...”“别只当成...”“不是...而是...”“简单讲...”这类模板句式。
- 第一行是否没有夸大事实。
